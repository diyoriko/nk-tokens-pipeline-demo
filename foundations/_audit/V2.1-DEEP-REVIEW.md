# V2.1 Deep Review — nk-tokens vs Figma SDS (2026-06-10)

Method: full SDS dump (community file, both Light/Dark modes, text+effect styles) vs our
tokens.json; OKLab lightness analysis of every ramp; WCAG contrast sweep of semantic pairs
beyond the CI contract. Status: review only — fixes land as one v2.1 batch after sign-off.

## P0 — real defects, fix before announcing

### 1. grey/100 too close to white → surfaces don't read
`base/secondary` (= grey/100 `#FAFAFA`) vs white: ΔL* 1.7. SDS Gray/100 = `#F5F5F5` (ΔL* 3.5).
Same semantic topology, double the contrast. **Fix: redefine grey/100 → #F5F5F5** (SDS value).
One hex, no renumber, aliases carry it everywhere. Legacy label `= Alabaster` → `≈ Alabaster`.

### 2. border/default fails non-text contrast on interactive components
`border/default/default` = grey/300 `#B9B9B9` → 1.96:1 on white (WCAG 1.4.11 needs 3:1 for
input boundaries). SDS is worse (`#D9D9D9`, 1.48) — not a model here. Divider/card hairlines
may stay decorative, but inputs need a compliant border.
**Fix: add `border/default/strong` = grey/500 `#7F7F7F` (3.95:1) for form controls; document
`border/default/default` as decorative (dividers, cards). Add the pair to check-contrast.**

### 3. Statuses have no hover tokens
brand-* has primary/secondary/tertiary hovers; success/warning/danger/info have none —
interactive status chips/buttons can't hover. SDS covers hover on every tier.
**Fix: add `background/{status}/{bold,light}-hover` following the hover law (see 4).**

### 4. Hover law is split-brain (Diyor's criterion)
Current: every group hovers one step DARKER, except primary of light-main hues
(magenta/coral/green/orange/yellow) which hovers LIGHTER (/400). Two candidate laws:
- **(A) uniform darken** (SDS): always +1 step darker. Risk: light mains get muddy hovers
  (yellow/600 `#C6B321` olive, orange/600 brown) and on-primary hue/900 text on /600 must be
  re-checked (coral/magenta look marginal — exact ratios computed in the fix PR).
- **(B) "hover steps away from the on-colour"** (current behaviour, formalised): white-text
  hues darken, dark-text hues lighten. Contrast-safe by construction, visually asymmetric.
**Decision needed (Diyor). Either way the law gets written into variable descriptions and
status hovers follow it.**

### 5. yellow/green light branches are broken ramps
yellow 300 `#FEF08D` vs 400 `#FFF171`: ΔL* = **0.1** — a dead step (indistinguishable).
yellow has 5 steps above L*90 then a 15.7 cliff to /600; green similar (4 steps above L*90,
15.2 cliff). violet/blue/grey have the opposite skew (1 light step).
**Fix: re-smooth yellow+green 200–500 in OKLCH between anchors (Main /500 and /100 stay;
weak "≈" anchors may move). Status light backgrounds shift slightly — re-run contract.**

### 6. Effects gaps confirmed by SDS
- SDS Drop Shadow 200–500 are **2-layer** (ambient+key) — our single-layer was an accepted
  cut; now scheduled: build-tokens.mjs must compose multi-layer box-shadow (parts → composed).
- We have backdrop-blur tokens in code-only but **no Blur effect styles in Figma** (SDS has
  Overlay/Layer/Glass). Fix: create 4 BACKGROUND_BLUR styles via API (no font needed).

## P1 — gaps worth closing (quick)

7. **`text/utilities/on-overlay`** (white) — modal/scrim text has no token (SDS has it).
8. **`border/utilities/swatch`** (`#0000003D`-style alpha) — swatch/thumbnail hairline; the
   showcase currently fakes it with a luminance check. SDS has it.
9. **size/28 missing**: type scale jumps 24→32 (ratio 1.33, worst in scale); v1 had 28 for
   the subtitle zone. Also no hero size >60 (SDS goes to 72) — needed for landings?
10. **TS-local `border medium/large` tokens** (seen in plugin, not in repo) — discard in TS
    or formalise via PR; unsaved they'll leak into a future TS push.
11. **Letter-spacing**: only positive (0/0.5/1 px for small text). SDS uses −2…−3% on display
    sizes. Mikado at 48–60px may want a light negative. Taste call.
12. **Naming case (Diyor 2026-06-10): capitalize names in Figma** like SDS
    (`Background/Base/Primary`). Requires a coordinated rename: Figma API first
    (variable.name edit preserves IDs → bindings/aliases survive), then tokens.json keys +
    refs via a node migration script, then build scripts' path lookups, and the CSS/TS/Dart
    name transform must force lowercase so code names stay `--nk-color-background-…`
    (zero breaking change in code). Style names get the same pass (`Display/Lg`).
    Variant question: hyphens kept (`Brand-Violet/Primary-Hover`, safer) or SDS-style spaces
    (`Brand Violet/Primary Hover`, prettier).

## Accepted by design (documented, not bugs)

- `text/disabled` 2.78:1 / 1.93:1 — WCAG exempts inactive components; noted in descriptions.
- Inner shadows: 2 levels (subtle/pressed) vs SDS 6 — deliberate minimalism.
- Weights 400/700 only — Mikado licence scope (check resources/ for Medium before revisiting).
- Radius 9 steps vs SDS 4 — keep until components show real usage.
- blur/depth parts living in `size` — SDS does exactly the same (Depth/Blur in Size).
- Our hierarchy text colours darker than SDS (secondary 9:1 vs their 4.6:1) — stricter, fine.
- Statuses include `info` (SDS lacks it); 7 brand hues vs SDS single Brand — our model.

## Where we're already ahead of SDS

- CI contrast contract (70 pairs, every build) — SDS has nothing equivalent.
- Full Polaris ×25 spacing incl. 18 negatives (SDS: 13 + 5).
- code-only motion/z-index/opacity tokens (SDS has none).
- Dev Mode codeSyntax + usage-rule descriptions on variables.
